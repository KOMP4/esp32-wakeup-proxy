Аппаратно-программный комплекс для надежного удаленного включения домашнего сервера (Wake-on-LAN) из любой точки мира. Сочетает в себе умный HTTP-прокси на сервере (VPS) и физическое устройство на базе микроконтроллера ESP32-C3, подключенное напрямую к материнской плате целевого сервера.

## 💡 Какую проблему решает этот проект?

Стандартный протокол Wake-on-LAN (WoL) часто оказывается бессилен, если домашний сервер находится за сложным NAT провайдера или подключен через VPN-туннели (например, AmneziaWG), которые поднимаются *только после* загрузки ОС. 

Дополнительная проблема возникает с мобильными клиентами (например, приложением **Nextcloud**), использующими протокол WebDAV. Если сервер выключен, а прокси пытается запустить скрипт ожидания загрузки, мобильные клиенты зависают в бесконечном таймауте и выдают критические ошибки.

**ESP32 Power Stick решает это изящно:**
1. **Умная маршрутизация:** Python-прокси проверяет заголовок `Accept`. Если запрос идет от браузера, пользователь видит красивую HTML-страницу загрузки (с использованием Server-Sent Events). Если запрос идет от API/мобильного приложения, прокси моментально отдает честный `HTTP 503 Service Unavailable`, заставляя приложение корректно уйти в режим ожидания и повторить попытку позже без зависаний.
2. **Аппаратная независимость:** Команда на включение передается по независимому зашифрованному каналу прямо на материнскую плату.
3. **Безопасность:** Передача команды осуществляется по протоколу MQTT с использованием двустороннего TLS-шифрования (mTLS). Подключиться к брокеру без клиентских сертификатов невозможно.

## ⚙️ Архитектура

`Браузер/Приложение` ➔ `Python aiohttp Proxy (VPS)` ➔ `MQTT Broker (Docker, порт 8883)` ➔ `ESP32-C3 (за домашним роутером)` ➔ `Оптопара PC817C` ➔ `Материнская плата (PWR_SW)`

### Ключевые особенности
* **Оптимизация под одноядерный MCU:** В прошивке реализована жесткая очистка зависших TLS-сокетов (`espClient.stop()`) и обрезка таймаутов рукопожатия, что предотвращает утечки памяти и зависания mbedTLS на чипе ESP32-C3.
* **NAT Keep-Alive:** Настроен частый пинг MQTT брокера (15 секунд), чтобы домашний роутер не закрывал простаивающий TCP-туннель.
* **Асинхронность:** Серверная часть написана на `aiohttp` и `aiomqtt`, что обеспечивает нулевое блокирование при проксировании тяжелого трафика (с отключенным `auto_decompress` для корректной передачи gzip-пакетов).

## 🛠 Аппаратная часть (Hardware)

### Bill of Materials (BOM)
* **Микроконтроллер:** ESP32-C3 SuperMini (или любой другой ESP32).
* **Оптопара:** PC817C (для гальванической развязки и защиты цепей материнской платы).
* **Резистор:** 220 Ом.

### Схема подключения
| Пины ESP32 | Компонент | Пины материнской платы |
| :--- | :--- | :--- |
| `GPIO 4` | ➔ Резистор 220 Ом ➔ Анод (1) PC817C | |
| `GND` | ➔ Катод (2) PC817C | |
| | Эмиттер (3) PC817C ➔ | `PWR_SW` (Минус / Ground) |
| | Коллектор (4) PC817C ➔ | `PWR_SW` (Плюс / Power) |

> [!NOTE]
> **Настройки BIOS:** Для питания ESP32 от USB-портов выключенного компьютера, убедитесь, что в BIOS вашего сервера включена функция подачи питания на USB в режиме ожидания (обычно требует отключения режима *ErP Ready*).

## 🚀 Развертывание (Deployment)

### 1. Подготовка сертификатов (mTLS)
Сгенерируйте корневой сертификат (CA), сертификат сервера для Mosquitto и сертификат клиента для ESP32 и Python-скрипта. Положите их в папку `deploy/certs/`.

### 2. Запуск MQTT брокера
Перейдите в папку `deploy` и запустите брокер через Docker Compose. Он автоматически подхватит конфигурацию `mosquitto.conf`, оптимизированную для IoT (с параметром `max_keepalive 0`).

cd deploy
docker compose up -d
